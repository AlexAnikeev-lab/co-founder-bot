# Узкие места (заторы) при нагрузке 30+ / 400+ пользователей

Документ описывает выявленные узкие места и внесённые доработки для масштабирования бота.

---

## 1. База данных: несколько сессий на один запрос (критично)

**Где:** `middlewares/auth.py`, все `handlers/*.py`.

**Проблема:** На каждый апдейт (сообщение/колбэк) открывается **несколько** сессий БД:
- AuthMiddleware открывает сессию → `get_by_telegram_id` + часто `update(last_seen_at)` → сессия закрывается.
- Обработчик снова делает `async for session in get_session():` — открывается **вторая** сессия.

При 30+ одновременных пользователях пул (10 + 20 overflow = 30 соединений) быстро исчерпывается: 2+ сессии на запрос × 15 пользователей = 30+ соединений.

**Решение:** Middleware «одна сессия на запрос»: создаём сессию в начале цепочки, передаём в `data["session"]`, Auth и хендлеры используют её. Плюс увеличен пул соединений под пик 400 пользователей.

---

## 2. Обновление last_seen_at на каждое действие (критично)

**Где:** `middlewares/auth.py`.

**Проблема:** На **каждое** сообщение и колбэк выполняется `UserRepository.update(session, user, last_seen_at=...)` с `commit()`. Это лишние записи в БД и конкуренция за блокировки при высокой нагрузке.

**Решение:** Обновлять `last_seen_at` не чаще чем раз в 5 минут на пользователя (по таймстампу последнего обновления).

---

## 3. N+1 запросов при показе следующей карточки в разделе «Партнёры» (критично)

**Где:** `handlers/swipe.py` → `get_next_user_for_swipe()`.

**Проблема:**
- Загружаются **все** подходящие пользователи (без лимита).
- Для **каждого** кандидата вызывается `TestResultRepository.get_by_user_id(session, user.telegram_id)` — отдельный запрос к БД (N+1).
- При 100–400 пользователях в базе это 100–400 запросов на один «показ следующей карточки» + расчёт совместимости в цикле.

**Решение:** Один запрос за списком кандидатов с лимитом (например 50–100), один batch-запрос за всеми `TestResult` по `user_id` этих кандидатов. Расчёт совместимости в памяти по уже загруженным данным.

---

## 4. Throttling: общий словарь без синхронизации (средне)

**Где:** `middlewares/throttling.py` — `user_last_request: Dict[int, float]`.

**Проблема:** При высокой конкуренции проверка и запись времени не атомарны. Плюс неограниченный рост словаря при большом числе пользователей.

**Решение:** Использовать блокировку (asyncio.Lock) на пользователя (или общий lock для быстрой проверки). Ограничить размер словаря (например, LRU/очистка старых записей).

---

## 5. Удаление предыдущих сообщений перед каждой отправкой (средне)

**Где:** `middlewares/delete_previous.py` — перед каждым `send_message`/`send_photo` вызывается `_delete_previous_messages()`.

**Проблема:** Перед отправкой нового сообщения выполняется **последовательный** цикл `await bot.delete_message(...)` по всем сохранённым message_id. Несколько вызовов Telegram API подряд задерживают отправку ответа пользователю. При большом числе сообщений в «пачке» задержка заметна.

**Решение:** Не ждать завершения удалений перед отправкой: запускать удаление в фоне (`asyncio.create_task`) и сразу отправлять новое сообщение. Либо ограничить размер хранимой пачки (например, последние 3–5 сообщений).

---

## 6. Размер пула соединений БД (средне)

**Где:** `repositories/database.py` — `pool_size=10`, `max_overflow=20`.

**Проблема:** 30 соединений суммарно при 400 пользователях в пике недостаточно, если на запрос уходило несколько сессий (см. п. 1). После введения одной сессии на запрос пул можно подобрать под ожидаемое число одновременных запросов.

**Решение:** Увеличен пул (например 20 + 40 overflow) и при необходимости добавлены таймауты пула. Точные значения можно настроить по метрикам под прод.

---

## 7. Общие словари в памяти без ограничения размера (низко)

**Где:** `middlewares/throttling.py` (`user_last_request`), `middlewares/delete_previous.py` (`_last_message_ids`).

**Проблема:** При длительной работе и большом числе чатов словари растут без ограничений.

**Решение:** Для throttle — периодическая или LRU-очистка. Для delete_previous — ограничение числа хранимых message_id на чат (например, только последние 5–10).

---

## 8. Множественные последовательные запросы в одном хендлере (низко/средне)

**Где:** Различные хендлеры в `handlers/swipe.py`, `handlers/profile.py` и др.

**Проблема:** В одном обработчике несколько раз подряд вызываются `UserRepository.get_by_telegram_id(session, same_id)` или другие повторные чтения. Логика при одной сессии на запрос уже лучше (кеш сессии), но дублирующие вызовы можно сокращать (переиспользовать объект `user`).

**Рекомендация:** По возможности передавать уже загруженного `user` из `data["user"]` и не дергать БД повторно без необходимости.

---

## Итог приоритетов

| Приоритет | Узкое место | Статус |
|-----------|-------------|--------|
| Критично  | Несколько сессий на запрос | Исправлено (DbSessionMiddleware + одна сессия) |
| Критично  | last_seen_at на каждое действие | Исправлено (троттлинг 5 мин) |
| Критично  | N+1 в get_next_user_for_swipe | Исправлено (batch TestResult) |
| Средне    | Throttling без lock / рост словаря | Исправлено (lock, ограничение размера) |
| Средне    | Удаление сообщений блокирует отправку | Исправлено (удаление в фоне) |
| Средне    | Пул БД | Увеличен (20 + 40) |

После внедрения этих изменений бот рассчитан на пиковую нагрузку порядка 400+ пользователей при условии одной сессии на запрос и оптимизации тяжёлых сценариев (свайпы, last_seen).
